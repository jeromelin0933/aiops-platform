# 金融級 AIOps 智能運維平台 (AI-Powered Fintech AIOps Platform)

本專案旨在建構一套全自動化的智能維運（AIOps）平台。針對金融業高併發、低延遲的交易環境，系統能即時監控底層指標與日誌，並在故障發生時，於 1 分鐘內完成「異常偵測 → 智慧收斂 → RAG 知識檢索 → AI 根因分析（RCA）→ 生成排障 SOP」的自動化閉環流程，大幅降低系統平均修復時間（MTTR）。

---

## 1. 產品需求文件 (PRD)

### 1.1 專案背景與核心痛點

在微服務架構日益複雜的金融系統中，產線發生異常時，維運團隊（SRE/DevOps）往往面臨三大挑戰：

- **告警風暴（Alert Storm）**：單一核心節點失效引發成百上千個連鎖告警，導致運維人員遭遇告警疲勞（Alert Fatigue）。
- **排障耗時（High MTTR）**：仰賴人工跨系統撈取日誌（Log）與監控指標（Metrics）進行人工比對，難以在第一時間定位根本原因。
- **知識斷層**：關鍵排障 SOP 往往存在於資深工程師腦海中，或散落於非結構化的維修手冊與文件，事故當下無法即時檢索與應用。

### 1.2 產品願景

建構一個「全自動化虛擬 SRE 大腦」，將傳統人工排障流程徹底自動化。透過無監督學習精準偵測異常，並結合大語言模型（LLM）與檢索增強生成（RAG）技術，自動關聯內部知識庫，在戰情室儀表板輸出具備金融規章架構的排障建議與修復 SOP。

### 1.3 核心功能需求 (Functional Requirements)

#### Phase 1：基礎觀測與數據收集層

- **實時指標與日誌監控**：系統須具備即時聚合應用程式效能指標（如 p95 Latency、QPS、Error Rate）與應用程式日誌（Log）之能力。
- **災難情境模擬**：PoC 階段須具備高併發交易風暴模擬器，用以生成連鎖故障數據（如資料庫連線池耗盡 DB Connection Pool Exhausted 觸發之 Timeout）。

#### Phase 2：基於無監督學習的異常偵測層

- **動態基準線學習**：捨棄傳統靜態閾值告警，系統須利用無監督學習演算法，自動學習數據常態，識別多維度指標的離群值（Anomaly）。
- **串流偵測機制**：系統須以固定週期（如 15 秒）掃描觀測層數據流，一旦判定異常即刻觸發告警事件。

#### Phase 3：AI 根因分析與智慧收斂層

- **智慧告警收斂（Alert Deduplication）**：遭遇連鎖故障時，系統須啟用防護冷卻期機制。在首筆告警觸發 AI 分析後，自動於設定時間內（如 60 秒）收斂並攔截同事件引發的餘波告警，防止告警風暴與後端 AI 算力枯竭。
- **企業知識庫治理與 RAG 檢索**：當告警觸發，系統須對接企業內部維修手冊或 SOP 知識庫，透過文字切塊（Chunking）與向量化（Embedding）技術，精準檢索與該故障高度相關的技術文件。
- **結構化 RCA 報告生成**：AI 大腦須整合異常指標、真實日誌與 RAG 檢索出之 SOP 知識，透過推理引擎產出包含「故障摘要、根因假說、事故時間軸、具體修復步驟」之結構化（JSON）報告。
- **運維戰情室 Dashboard**：系統須提供即時監控儀表板，動態展示最新異常事件、核心指標圖表、智慧收斂狀態，並完整呈現 AI 產出之 RCA 報告與修復指引。

### 1.4 非功能性需求 (Non-Functional Requirements)

- **高可用降級防護 (Mock Fallback)**：當外部 AI 服務發生異常、網路斷線或觸發速率限制（429 Too Many Requests）時，系統須具備 Fallback 機制，自動切換至離線專家知識庫（Mock Mode），確保前端儀表板持續可用。
- **資安去識別化（Data Masking）**：考量金融資安，知識庫手冊與產線日誌在進入向量資料庫或大模型分析前，須建立去識別化機制，確保敏感交易資訊不外流。

## 2. 系統架構技術選型與使用理由 (Tech Stack & Rationale)

平台架構嚴謹切分為觀測、偵測、收斂、決策與輸出五大核心模組，各組件選型與技術職責如下：

### 2.1 Prometheus & Grafana Loki｜觀測層

**它是做什麼的**

Prometheus 負責時間序列指標數據（Metrics）的實時收集與儲存；Grafana Loki 則作為低延遲的日誌（Logs）聚合中樞。

**為什麼使用它**

它們是雲原生維運（Cloud Native Ops）的業界標準。Prometheus 的拉取機制能確保高併發下的指標穩定度；Loki 採用與 Prometheus 相同的標籤（Label）設計，這讓系統在 Phase 3 進行「指標與日誌的實時時間軸關聯比對」時，能達到極高的查詢效率。

---

### 2.2 Scikit-learn — 隔離森林演算法 (Isolation Forest)｜偵測層

**它是做什麼的**

作為無監督機器學習的核心偵測引擎，負責掃描多維度指標特徵向量，判定是否為異常離群點。

**為什麼使用它**

金融系統的故障往往不是單一指標超標（如 CPU 飆高），而是延遲增加與錯誤率同時上升的多維度異常。隔離森林（Isolation Forest）在處理多維度、不需要預先標記（Unlabeled）的串流數據時效率極高，且運算資源消耗低，非常適合作為產線第一線的實時異常偵測引擎。

---

### 2.3 Rule-based Correlation Engine｜智慧收斂層

**它是做什麼的**

實作時間視窗冷卻鎖邏輯。當 Phase 2 產生大量重複性或連鎖性的告警時，此引擎負責攔截餘波，並將其歸納收斂至同一個根因事件中。

**為什麼使用它**

在真實金融海嘯中，幾秒內會湧入上百筆同源警報。如果不做收斂，後端 LLM 將會在 1 分鐘內因為併發呼叫而觸發 429 錯誤而直接癱瘓。此機制是保護企業 API 算力成本、確保 AIOps 系統高可用性的「保險絲」。

---

### 2.4 LangChain & 地端向量資料庫 (Vector Store)｜決策層 — 知識檢索

**它是做什麼的**

利用 LangChain 技術框架進行企業內部維修手冊（PDF/DOCX）的資料清洗、文本切塊（Chunking）與嵌入（Embedding），並儲存於地端向量資料庫中（如 ChromaDB / Pinecone），實現 RAG（檢索增強生成）。

**為什麼使用它**

通用大模型不具備企業內部的基礎架構與特定資料庫修復 SOP。透過 RAG 技術，能讓 AI 大腦在生成排障建議時，強制綁定內部合規的技術文件。使用地端向量資料庫則是基於金融資安考量，確保內部文件不外流。

---

### 2.5 LLM API / 地端模型 Fallback 引擎｜決策層 — 邏輯大腦

**它是做什麼的**

負責接收異常指標、真實錯誤日誌與 RAG 檢索出的 SOP，進行語意理解與結構化（JSON）邏輯推理，產出核心 RCA 報告。

**為什麼使用它**

傳統維運自動化腳本只能做到「若 A 則 B」，無法理解雜亂無章的非結構化日誌（Log text）。大語言模型具備強大的語意理解與上下文推理能力，能像資深工程師一樣「讀懂」犯罪現場。架構設計上支援「API 與地端模型雙軌 Fallback」，可根據企業資安需求靈活動態切換。

---

### 2.6 FastAPI Web Server & Dashboard｜輸出層

**它是做什麼的**

負責提供監聽與告警串接的 API 端點，並即時渲染具備實時觀測數據圖表與 AI RCA 報告的運維戰情室儀表板。

**為什麼使用它**

FastAPI 具備極高的非同步（Async/Await）效能，基於 Starlette，效能甚至可媲美 Node.js 和 Go，非常適合用來處理 AIOps 這種需要實時接收底層警報與日誌的高併發場景。同時，客製化 Dashboard 解決了傳統 Grafana 原生面板難以清晰、結構化呈現大模型長文本、故障時間軸與複雜 SOP 排障步驟的痛點。
